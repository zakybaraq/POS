# Phase 5: Stock Synchronization Verification

## Context

The stock synchronization logic for order completion has been implemented in the codebase. This phase focuses on:
1. Verifying the existing implementation meets the requirements
2. Adding comprehensive tests for edge cases
3. Ensuring transaction rollback on failures
4. Validating stock movement logging

### Current Implementation Status

**Stock decrementation occurs in:**
- `src/repositories/order.ts`: `completeOrder()` and `completeOrderWithPayment()`
- `src/repositories/inventory.ts`: `decrementStockForOrderTx()`

**Key behaviors:**
- Stock is only decremented when order reaches "completed" status
- Decrementation happens within the same database transaction as order status update
- Duplicate prevention via `existingMovements` check
- Stock refund on cancellation via `refundStockForOrderTx()`
- Full stock movement audit trail in `stockMovements` table

### Decisions from CONTEXT.md
- **D-01**: Stock decremented ONLY on order completion (already implemented)
- **D-02**: Data consistency prioritized over performance (already implemented via transactions)

## Task 2: Create Comprehensive Edge Case Tests

**Files:**
- `test/integration/stock-synchronization.test.ts` (new)

**Action:**
Create a comprehensive test suite covering all edge cases for stock synchronization:

**Test Cases to Implement:**

1. **Basic Flow:**
   - Order creation → Add items → Complete with payment → Stock decremented
   - Verify ingredient stock reduced by recipe amount × quantity

2. **Duplicate Prevention:**
   - Complete order once → Stock decremented
   - Attempt to complete same order again → No duplicate decrement
   - Verify `existingMovements` check works

3. **Transaction Rollback:**
   - Start completion → Force error during stock decrement
   - Verify order status NOT updated (remains active)
   - Verify stock unchanged

4. **Insufficient Stock:**
   - Create order requiring more stock than available
   - Attempt completion → Should throw error
   - Verify order NOT completed
   - Verify stock unchanged

5. **Multi-Ingredient Menu:**
   - Menu with 3+ ingredients
   - Complete order → All ingredients decremented
   - Verify each ingredient's stock correctly reduced

6. **Order Cancellation (Completed):**
   - Complete order → Stock decremented
   - Cancel order → Stock refunded
   - Verify refund stock movement created

7. **Order Cancellation (Active):**
   - Active order (not completed)
   - Cancel → No stock changes (nothing to refund)

8. **Finish Without Payment:**
   - Use `finishOrderWithoutPayment()`
   - Verify stock NOT decremented (per design)
   - Verify order completed

9. **Idempotency Under Concurrency:**
   - Simulate simultaneous completion attempts
   - Verify only one succeeds in decrementing stock

**Test Setup Requirements:**
- Create test ingredients with known stock levels
- Create test menu with recipes linking to ingredients
- Use database transactions for isolation
- Clean up test data after each test

**Acceptance Criteria:**
- [ ] All 9 test cases implemented
- [ ] Tests run successfully: `bun test test/integration/stock-synchronization.test.ts`
- [ ] Edge case coverage documented
